**Project:** PC Build Recommendation System (Web-Based)
**Milestone:** 3 — Dataset Preprocessing
**Group Members:** Umer Daraz, Habib, Kashif

## 1. Overview

This document describes how data enters, moves through, and leaves the database of the PC Build Recommendation System. It is written specifically for this project — the entities (`User`, `Build_Request`, `Component`, `Recommended_Build`, `Build_Component`) and the recommendation logic described below are unique to our system. The flow follows the normalized schema finalized in Milestone 2.

---

## 2. Data Sources (where data enters the system)

Data enters the system through three distinct entry points:

**A. User signup form (frontend → `User` table).**
When a visitor registers, the Flask backend accepts `full_name`, `email`, and `password` from the HTML form, hashes the password using bcrypt, assigns the default role `user`, and inserts a new row into `User`. The `user_id` is auto-generated by MySQL.

**B. Build request form (frontend → `Build_Request` table).**
After login, a user opens the build request page and submits two inputs: a numeric `budget` (PKR / USD) and a `purpose` chosen from a fixed dropdown (Gaming, Programming, Video Editing, General Use, 3D Rendering, Streaming). The backend reads these, attaches the logged-in user's `user_id` and the current `request_date`, and inserts a row into `Build_Request`.

**C. Admin panel for component catalog (admin frontend → `Component` table).**
Administrators (users with `role = 'admin'`) can add, edit, or delete hardware components through a protected admin page. Each entry stores `component_name`, `component_type` (CPU, GPU, RAM, Motherboard, Storage, PSU, Case, Cooler), `brand`, `price`, `specifications`, and `compatibility_info`. This is the only entry point that writes to the `Component` table; regular users have read-only access to it.

For Milestone 3, all three of these inputs are populated using the synthetic CSV files we generated (60 users, 66 components, 80 build requests). In production, the CSVs would be replaced by live form submissions and live admin updates.

## 3. Internal Processing (how data moves through the database)

Once a `Build_Request` row exists, the recommendation engine runs on the backend. The flow is:

**Step 1 — Read the request.**
The engine fetches the new row from `Build_Request` to obtain `budget` and `purpose`.

**Step 2 — Query the catalog.**
The engine queries `Component` to pull all available hardware grouped by `component_type`. The query is filtered and ranked using two project-specific rules:
- *Budget allocation:* the total budget is split across component types using fixed percentages calibrated for PC builds — CPU ~22%, GPU ~32%, RAM ~8%, Motherboard ~12%, Storage ~8%, PSU ~8%, Case ~5%, Cooler ~5%. Each part is then chosen from `Component` rows whose `price` fits its slice.
- *Compatibility:* the engine inspects `compatibility_info` to match CPU socket with motherboard socket (e.g., AM5 CPUs only with AM5 boards) and to match RAM generation with the chosen motherboard (DDR4 vs DDR5). Incompatible combinations are filtered out before the final selection.

**Step 3 — Adjust by purpose.**
The engine biases choices toward the request's `purpose`. Gaming requests favor stronger GPUs; Video Editing and 3D Rendering favor higher-core-count CPUs and larger RAM (sometimes with `quantity = 2` to give 64 GB); Programming and General Use stay with mid-range parts. Larger budgets occasionally trigger a second storage device (NVMe SSD + secondary HDD).

**Step 4 — Write the recommended build.**
A new row is inserted into `Recommended_Build` with `total_cost` (sum of every chosen part's price × quantity), `recommendation_date`, and the originating `request_id`.

**Step 5 — Write the build–component links.**
For each component included in the build, the engine inserts one row into `Build_Component` containing the new `build_id`, the `component_id`, and a `quantity`. This junction table is what resolves the many-to-many relationship between `Recommended_Build` and `Component` (the change made during Milestone 2 normalization).

**Step 6 — Optional second recommendation.**
About 30% of requests trigger a second `Recommended_Build` row (an alternative build at a different price point), giving the user two options to compare. Both builds share the same `request_id`.

### Table dependency order

The dependency direction (which table needs which) is:

```
User  ──►  Build_Request  ──►  Recommended_Build  ──►  Build_Component  ◄──  Component
```

Reading left-to-right: a `Build_Request` cannot exist without a `User`, a `Recommended_Build` cannot exist without a `Build_Request`, and a `Build_Component` row needs both an existing `Recommended_Build` and an existing `Component`. This same order is the loading order for Milestone 5 — CSVs must be imported in this sequence or foreign-key checks will fail.

## 4. Outputs (what comes out of the system)

The system produces three kinds of output:

**A. Build recommendation page (primary user-facing output).**
After submitting a request, the user is redirected to a results page. The backend joins `Recommended_Build`, `Build_Component`, and `Component` on `build_id` and `component_id` to produce a list of parts per recommended build, with each row showing `component_name`, `brand`, `price`, `quantity`, and the running `total_cost`. If two builds were generated for the request, both are shown side-by-side.

**B. User history page.**
A logged-in user can view all their past requests. The query joins `User → Build_Request → Recommended_Build → Build_Component → Component` filtered by `user_id`, and returns every build the user ever received, sorted by `recommendation_date` descending.

**C. Admin reports.**
The admin panel runs aggregate queries against the same tables, for example:
- Most-recommended components → `GROUP BY component_id` on `Build_Component` with a count, joined to `Component` for names.
- Average budget by purpose → `GROUP BY purpose` on `Build_Request` with `AVG(budget)`.
- Builds per month → `GROUP BY MONTH(recommendation_date)` on `Recommended_Build`.

These outputs are read-only views; they do not write back to any table.

## 6. Synthetic Dataset Summary (Milestone 3 deliverable)

For Milestone 3 we generated synthetic data using Python with the `Faker` library. Random seeds are fixed (`Faker.seed(42)`, `random.seed(42)`) so the data is reproducible. Five CSV files were exported, one per table:

| Table              | Rows | Notes |
|--------------------|-----:|-------|
| `User.csv`              | 60  | 5 admins + 55 regular users; all emails unique; passwords stored as bcrypt-style placeholder hashes |
| `Component.csv`         | 66  | Real hardware names across 8 component types (CPU, GPU, RAM, Motherboard, Storage, PSU, Case, Cooler); realistic price ranges |
| `Build_Request.csv`     | 80  | Budgets correlated with purpose; dates spread Jan 2025 – May 2026; every `user_id` references a real regular user |
| `Recommended_Build.csv` | 106 | ~30% of requests have a second alternative build; `total_cost` equals the exact sum of its component rows |
| `Build_Component.csv`   | 848 | Junction rows; every `build_id` and `component_id` references a real parent row; quantities are 1 by default, 2 for some high-budget RAM kits |

**Verification performed before export:**
- All foreign-key references resolve to existing parent rows (0 orphan rows across all tables).
- All primary keys are unique within each table.
- All emails in `User` are unique.
- For every `Recommended_Build` row, `total_cost` exactly equals the sum of `price × quantity` over the matching `Build_Component` rows.

The five CSV files are committed to the repository alongside this document, ready to be loaded into MySQL in Milestone 5.
